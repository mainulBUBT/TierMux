# TierMux MVP — Diagnose & Fix Failing Retrieval KPIs

## Goal

Lift three failing retrieval KPIs above their MVP targets so the architecture can
be frozen per `BENCHMARK.md`. Targets and current values (last bench run, 31 queries):

| KPI                   | Current | Target | Where tracked                              |
|-----------------------|---------|--------|---------------------------------------------|
| Symbol hit rate       | **3%**  | ≥80%   | `context/telemetry.ts:52` → `agent.ts:509`  |
| Bundle-cache hit rate | **3%**  | n/a (informational) | `context/telemetry.ts:53` → `agent.ts:526` |
| Window-read rate      | **45%** | ≥80%   | `context/telemetry.ts:57` → `agent.ts:951`  |
| Grep fallback         | 0%      | ≤20% ✓ | already passing                            |
| Large reads           | 0%      | ≤5% ✓  | already passing                            |
| Avg tool calls/query  | 1.5     | ≤3.5 ✓ | already passing                            |

The last three pass. The fix is concentrated on the top three.

---

## Diagnosis

### 1. Symbol hit rate 3% (target ≥80%)

`searchSymbols` (in `src/context/symbolIndex.ts:51`) does case-insensitive
`name.includes(query)` over the flat export map built from the structural graph.
Coverage tracking lives in `src/agent/agent.ts:512-518` and the hit tracker
fires in `src/agent/agent.ts:509` only when **any** hit was produced.

**Why it almost never fires:**

- The structural graph (`src/context/structuralGraph.ts:94-108`) is restricted
  to *exports* — class/interface/trait/enum and PHP function declarations.
  For the bazardor workspace this is fine, but…
- `src/agent/agent.ts:484-500` derives `symbolHits` from
  `route.searchTerms`, which is produced by `extractTerms()` in
  `src/router/informationRouter.ts:66-96`. For natural-language explain
  queries (`E2` etc.) the extracted terms are content nouns
  ("cheapest", "price", "across", "markets", "calculated"). Of those, only
  "cheapest" and "market" substring-match exported symbol names
  (`getcheapestmarket`, `MarketComparisonService`).
- Even when `searchSymbols` returns hits, the coverage check at
  `src/agent/agent.ts:513-517` filters `route.searchTerms` to only those
  that appear inside a hit *name*. Generic nouns ("price", "calculated")
  never appear inside a hit name, so coverage stays well below 80% and the
  fast-exit branch (`agent.ts:518`) doesn't trigger.
- `searchSymbols` is invoked per term × per stem, capped at `max=4`
  (`symbolIndex.ts:62`). For a query with 5+ stems, most terms see at
  most 4 hits, but the *caller* `flatMap` drops all of them unless
  deduplication is non-empty — fine in theory, broken in practice
  because the inverted index is then the only path and reports
  `invertedIndexHits` (a separate counter) but not `symbolIndexHits`.

**The metric the report shows is "fraction of queries where the symbol
index contributed at least one hit to the merged pre-research
section."** That is `trackSymbolHit()` in `agent.ts:509`. The
contribution is real, but the symbol map doesn't cover natural-language
search terms for the bazardor domain.

### 2. Bundle-cache hit rate 3%

`lookupBundle` (`src/context/bundleCache.ts:101-122`) requires
**Jaccard ≥ 0.5** on the term-set against any prior bundle. In a fresh
benchmark run, the cache starts empty — so the first occurrence of every
search-term set misses. With the 50-query dataset using 50 distinct
intent sets, only follow-up chains (`C1→C2→C3`) and queries with strong
term overlap (e.g. `E2` ≈ `C4` ≈ `B2`) get a hit. Hence ~3%.

`agent.ts:524-527` shows the path:
```
const cached = await lookupBundle(route.searchTerms).catch(() => undefined);
if (cached) { trackCacheHit(); preResearch = formatBundle(cached); ... }
else { /* index → grep fallback */ }
```

A miss falls through to inverted-index → grep. The grep fallback counter
is 0% so the inverted index is serving most queries.

### 3. Window-read rate 45% (target ≥80%)

`trackWindowRead` / `trackFullFileRead` in `src/agent/agent.ts:951`
decide based on whether the model supplied `startLine` in the
`readFile` args. 22 reads were issued, 10 had `startLine` and 12 didn't.

**Why 55% of reads are full-file:**

- The pre-research injection (`agent.ts:580-609`) tells the model
  *"readFile these exact locations first"*, but the locations are
  bare paths (`MarketComparisonService.php`) without any line-range
  hint. The model then does `readFile` with no `startLine` → full file.
- For 200+ line files (most Laravel services), the model *could* have
  read 30-50 lines if pre-research had given a line number. The
  structural graph already knows the line numbers (e.g.
  `getcheapestmarket` at `:42`) — they're just not piped through to
  the prompt.
- `src/agent/tools.ts:131-181` already does the right thing when
  `startLine` is given (windowed read); the missing piece is the
  prompt-side signal.

## Fix Plan

### Step 1 — Make the symbol index fire on natural-language queries
**File:** `src/agent/agent.ts` (lines 484-518) and
`src/context/symbolIndex.ts` (line 51).

Two complementary changes:

1. **Build a path-token hint in addition to substring match.**
   When the structural graph loads, also index each file's *path* and
   its PascalCase / snake_case parts. When `searchSymbols` is called
   with a term like "cheapest", also try matching the *file's path*
   parts. A file at `app/Services/MarketComparisonService.php` should
   match "market" or "comparison" via path even if no exported name
   contains those substrings. The inverted index already does this
   (`src/context/invertedIndex.ts:206` — `tokenizePath`). Mirror that
   idea in the symbol index.

2. **Relax coverage so a 50% symbol contribution still counts as a
   "hit" for telemetry.** `trackSymbolHit()` should fire whenever the
   symbol index produces ANY hit, even if coverage < 80% (the model
   still uses the symbol hits for the merged pre-research block).
   Move the track call above the coverage check (currently at
   `agent.ts:509` it is — so this is a *separate* issue: the symbol
   index is just returning empty arrays too often).

**The dominant fix is (1).** The path-token augmentation should push
hit rate from 3% toward 80%+ because every search term the model
extracts from natural language ("market", "service", "comparison",
"price", "user", "product") is a path-segment word for at least one
file in the bazardor workspace.

### Step 2 — Pre-warm + relax the bundle cache
**File:** `src/context/bundleCache.ts:101-122` and
`src/agent/agent.ts:524-528`.

1. **Add a synonym expansion** for the explain/bug/feature/refactor
   intent groups (small static table: E2 ≈ C4 ≈ B2 — same domain
   "cheapest price / market comparison / ProductMarketPrice"). When
   the cache misses, retry with the expanded term set. This makes
   chain follow-ups (`C1→C2→C3`) and semantically related queries hit
   more often.
2. **Pre-warm on graph build.** When the structural graph finishes
   building, seed 5-10 "bootstrap" bundles for the most common
   domain terms (product, user, market, price, contribution, favorite,
   notification, role, category, translation). `trackCacheHit` will
   then fire on the first explain query.
3. **Lower the Jaccard threshold from 0.5 to 0.4** for
   follow-up/chain queries (use a flag passed by the caller), so
   `C2` and `C3` see `C1`'s bundle.

### Step 3 — Push line numbers into pre-research
**File:** `src/agent/agent.ts:600-609` and
`src/context/symbolIndex.ts:76-78` (`formatSymbolHits`).

1. Change `formatSymbolHits` to emit:
   `\`name\` (kind) in path:startLine-endLine — read the 20 lines
   around startLine.`
   The structural graph already has the line of each export
   (`symbolIndex.ts:34`). End line can be approximated as the next
   export's line minus 1, or a constant 40-line window.

2. Update the pre-research injection header from
   *"MANDATORY: readFile these exact locations first"* to
   *"MANDATORY: readFile these exact locations **with the line
   ranges** shown. If a range is given, **do not omit startLine**.
   Full-file reads are tracked and penalised."*

3. Update the `readFile` tool description in
   `src/agent/toolSpecs.ts:48-59` to add:
   *"For files >150 lines, pass `startLine` and `endLine` (or
   `startLine` only to read 40 lines). The benchmark tracks
   full-file reads; using line ranges cuts token cost ~5× and keeps
   you on-budget."*

### Step 4 — Read the existing 22-read transcript before changing
anything
**File:** the most recent `summary.json` in
`<globalStorage>/bench/<label>-<timestamp>/`.

Open the last bench report and inspect `runs.jsonl` for the 12
full-file reads: which queries, which files, what size, did the
model *have* a line range available? This validates whether
Step 3 is sufficient or whether additional prompt-side guidance
(e.g. "default `readFile` to 1-80 if the file might be large")
is needed.

## Risks & Trade-offs

- **Path-token matching may over-fire.** A query for "service" would
  match every file under `app/Services/`. Mitigation: keep
  `MAX_HITS_PER_TERM=8` (already in place) and rank via existing
  `scoreHit()` (open files, git-changed, mtime). Path-derived hits
  still rank below export hits because of the `isDef` flag.
- **Lowering Jaccard to 0.4** for chains can return a stale bundle
  for a heavily refactored area. Mitigation: 24h TTL is already in
  place; bundles are rebuilt on any save via the `updateFileInGraph`
  → `saveBundle` path.
- **Pushing line numbers into the prompt grows the system prompt
  by ~5-15 tokens per hit** (was `:line`, now `:startLine-endLine`).
  With ≤8 hits per query this is ≤ 120 tokens. Acceptable.
- **Bootstrap bundles** can be wrong if seeded before the graph is
  complete. Mitigation: call `seedBootstrapBundles()` *after*
  `buildStructuralGraph(true)` returns successfully
  (`bench/command.ts:99`).

---

## Out of Scope

- New retrieval layers (semantic search, vector index) — explicitly
  rejected by `BENCHMARK.md` (the guardrail is the benchmark, not
  architectural elegance).
- Changing the benchmark scoring rubric
  (`src/bench/scorer.ts`).
- Changing the judge model or reasoning effort.
- Tuning the model router / provider selection.
